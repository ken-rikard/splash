<objective>
**Plan 02: Express Server + Live Data Integration**

Create the Express HTTP server that imports the Phase 1 scraper engine and exposes river data via REST (`/api/rivers`, `/api/rivers/:id`) and Server-Sent Events (`/api/events`). Update the React hooks to fetch from the real API and subscribe to SSE for live updates. After this plan, the dashboard displays real data from the scraper.

**Purpose:** Bridge Phase 1's scraper engine to the Phase 2 frontend. The server runs in the same Node.js process as the engine, reading directly from FlowStore memory and bridging ScraperEventBus events to browser clients via SSE.

**Output:** Express server at project root, updated hooks, working real-data dashboard.
</objective>

## Web Security Notes

### XSS
React JSX escaping mitigates stored and reflected XSS by default. No `dangerouslySetInnerHTML` is used. The API returns only RiverData payloads (numeric levels, string station names, dates) — no user-generated or renderable HTML content. Localhost-only, single-user, and read-only data flow further reduces risk to acceptable level for v1.

### CSRF
No mutation endpoints exist in Phase 2 — SSE is read-only (GET), REST endpoints are GET-only. No POST/PUT/DELETE routes are defined. CSRF is not applicable for this phase.

### Content Security Policy (CSP)
No CSP header is configured in Phase 2. The app runs on localhost only and serves content from the same origin via Vite proxy (dev) or Express static (prod). Acceptable for v1 localhost usage. For production deployment, consider adding a `default-src 'self'` policy with `connect-src 'self'` to allow API calls.
</threat_model>
