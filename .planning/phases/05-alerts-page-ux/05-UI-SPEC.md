## Interaction Contract

### State Machines

**AlertProvider SSE connection lifecycle:**
```
[lazy] вҶ’ loading (initial REST fetch pending)
       вҶ’ connected (REST fetch resolved OR any SSE event received)
       вҶ’ error (fetch failed OR SSE error event)
       вҶ’ [reconnects via EventSource auto-reconnect вҶ’ connected]
```

**AlertCard visibility lifecycle:**
```
activeAlert received via SSE/REST вҶ’ rendered in list
  вҶ’ user clicks dismiss вҶ’ dismissedAt stored in localStorage (riverId вҶ’ timestamp)
  вҶ’ alert hidden from active list (filtered by undismissedAlerts)
  вҶ’ if same river re-triggers (newer triggeredAt > dismissedAt) вҶ’ reappears
  вҶ’ if alert-resolve SSE received вҶ’ moved to resolvedAlerts list
```

### Interaction Behaviors

| Element | Interaction | Behavior |
|---------|------------|----------|
| AlertCard dismiss button (`X`) | Click | Calls `dismissAlert(riverId)` вҶ’ adds `{riverId: Date.now()}` to localStorage вҶ’ alert removed from `alerts` list вҶ’ `count` decreases |
| AlertCard "View River Details" | Click | React Router navigation to `/river/:id` |
| AlertCard dismiss button visibility | Hover | `opacity-0 group-hover:opacity-100` вҖ” visible on card hover only |
| AlertCard | Mount | `animate-fade-in-up` with staggered `animationDelay: index * 80ms` (matches RiverCard pattern) |
| AlertCard border | Hover | `border-danger-4/20` вҶ’ `hover:border-danger-4/40` (subtle glow on hover) |
| Resolved `<details>` | Click/toggle | Collapsible (native `<details>` element) вҖ” shows recently resolved alerts |
| NavBar alert badge | Dynamic | `count` read from `useAlerts()` context; label updates to `Alerts ({count})` when `count > 0` |
| NavBar /alerts link | Click | React Router navigation to `/alerts` |
| SSE alert-trigger event | Received | `useAlerts` handler: merges with existing state (replace by `riverId`), re-parses `triggeredAt` from string to Date, re-checks dismissed state |
| SSE alert-resolve event | Received | `useAlerts` handler: removes from active alerts, adds to `resolvedAlerts` list |
| Initial REST merge | Page load | Merges SSE additions that server doesn't know about (race condition guard per RESEARCH Pitfall 1) |
| Cross-tab dismissal sync | вҖ” | Deferred to v2 (NOT in Phase 5). `BroadcastChannel` and `storage` event listener are NOT implemented per Anti-Pattern: "Alert dismissal syncs across tabs via BroadcastChannel API" is a Future Requirement. |

## Error Handling

| Condition | UX |
|-----------|-----|
| SSE connection fails / REST fetch fails | `status: 'error'` вҶ’ AlertsPage shows ErrorState: "Live updates unavailable / Could not connect to the river monitoring service" |
| Initial REST fetch succeeds, SSE later disconnects | Existing alerts remain visible; `status: 'connected'` вҶ’ SSE auto-reconnect (EventSource handles this); no new alerts until reconnection. No stale warning needed (different from river data вҖ” alerts don't get stale) |
| Server restarts (active alert state lost) | Client alerts become stale. On next scrape, server re-evaluates and `alert-trigger` events re-fire. Client checks `triggeredAt > dismissedAt` вҖ” if dismissal was before server restart, alerts reappear (correct behavior вҖ” new trigger event) |
| localStorage setItem throws (quota exceeded, private browsing) | `console.warn('Failed to persist alert dismissals:', err)` вҖ” dismissal works in-memory only; dismissed alerts reappear on page refresh |
| `JSON.parse` of SSE data fails | Error is caught by `try` block in EventSource message handler; malformed events are ignored (not fatal to connection) |
| RiverDetailPage: fetch config for non-existent river | `useAlertConfig` returns `config: null` вҖ” no alert section displayed (same as "no config") |
| RiverDetailPage: network error fetching config | `useAlertConfig` returns `config: null`, `loading: false` вҖ” no alert section displayed. User sees no indication of failure. Acceptable for v1.1 (non-critical info) |
