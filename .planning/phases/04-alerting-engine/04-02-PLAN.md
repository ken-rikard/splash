Implement alert evaluation logic, wire it into the scrape cycle, and expose active alert state via REST.

**Purpose:** Alerts must be evaluated automatically each time new river data arrives, and users need to query which alerts are currently active. This plan adds the threshold comparison engine, hooks it into the existing data-update event, and exposes the active alert state via a GET endpoint.

</objective>

    **Step B — Implement methods in `src/core/alert-engine.ts`:**

    Add the following methods to the existing `AlertEngine` class:

    `getActiveAlerts(): ActiveAlert[]`:
    - Returns `Array.from(this.activeAlerts.values())`

    `getActiveAlert(riverId: string): ActiveAlert | undefined`:
    - Returns `this.activeAlerts.get(riverId)`

    `evaluate(rivers: RiverData[]): void`:
    - Iterate over each `river` in `rivers`:
      1. `const config = this.configs.get(river.id)`
      2. If `!config || !config.enabled`: delete `this.activeAlerts.delete(river.id)`, continue
      3. If `river.currentLevel === null`: continue (skip evaluation)
      4. `const isTriggered = this.isThresholdExceeded(config, river)`
      5. `const hasActive = this.activeAlerts.has(river.id)`
      6. If `isTriggered && !hasActive`: create a new `ActiveAlert` with:
         - `riverId: river.id`
         - `config` (the config object)
         - `threshold: this.resolveThreshold(config)`
         - `currentValue: river.currentLevel`
         - `alertLevel: river.alertLevel`
         - `triggeredAt: new Date()`
         - `snapshot: { ...river }` (shallow copy)
         - Set on map: `this.activeAlerts.set(river.id, {...})`
      7. Else if `!isTriggered && hasActive`: `this.activeAlerts.delete(river.id)`
      8. (isTriggered && hasActive → already triggered, no update needed)
      9. (!isTriggered && !hasActive → nothing to do)

    `private isThresholdExceeded(config: AlertConfig, river: RiverData): boolean`:
    - If `config.type === 'level'`: return `config.level !== undefined && river.alertLevel >= config.level`
    - If numeric: return `config.customValue !== undefined && river.currentLevel > config.customValue`
    - The `>=` for level and `>` for numeric match the comparison semantics from RESEARCH.md.

    `private resolveThreshold(config: AlertConfig): number`:
    - If numeric: return `config.customValue!`
    - If level: return `config.level!`

    `private isConfigInactive(config: AlertConfig): boolean`:
    - If level: return `config.level === undefined`
    - If numeric: return `config.customValue === undefined`

    **Step C — Update `setConfig` to call `isConfigInactive`:**
    In the existing `setConfig` method, after `this.configs.set(...)`, add:

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-04 | Tampering | evaluate() | accept | Pure synchronous comparison of trusted internal data; no external input enters evaluate(). RiverData comes from ScraperEngine's fetch cycle — already validated |
| T-04-05 | Information Disclosure | GET /api/alerts/active | accept | Returns active alert state (river ID, threshold, numeric values, timestamp). No PII. Acceptable for v1.1 single-user deployment |
| T-04-06 | Denial of Service | evaluate() | accept | O(n) scan over configs Map per river — max ~300 rivers in dataset. evaluate() is synchronous and completes in microseconds. No DoS vector |
