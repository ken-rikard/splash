</objective>

  <action>
    **Step A — Add types to `src/core/types.ts`:**

    Add two new interfaces after `ScraperStatus` (end of file):

    1. `AlertConfig`:
       - `riverId: string` — compound key `"source:stationId"` format
       - `type: 'level' | 'numeric'` — mutually exclusive threshold type
       - `level?: AlertLevel` — required when `type === 'level'`; danger level 1-5
       - `customValue?: number` — required when `type === 'numeric'`; flow threshold in m³/s
       - `enabled: boolean` — whether this config is actively evaluated

    2. `ActiveAlert`:
       - `riverId: string`
       - `config: AlertConfig` — the config that triggered this alert
       - `threshold: number` — resolved threshold value that was crossed
       - `currentValue: number` — river flow at evaluation time
       - `alertLevel: AlertLevel` — river's danger level at evaluation time
       - `triggeredAt: Date` — when the alert first triggered
       - `snapshot: RiverData` — snapshot of RiverData that triggered the alert

    Follow the exact JSDoc commenting style from types.ts (one-line /** */ above the property).

    **Step B — Create `src/core/alert-engine.ts`:**

    Class `AlertEngine` with:
    - Private `configs = new Map<string, AlertConfig>()` (riverId → config)
    - Private `activeAlerts = new Map<string, ActiveAlert>()` (placeholder — active alert methods added in Plan 02)

    Methods:
    - `setConfig(config: AlertConfig): AlertConfig` — upsert by `config.riverId`. If `!config.enabled` or the config is inactive (level undefined when type='level', customValue undefined when type='numeric'), also delete from `activeAlerts`. Return the config.
    - `getConfig(riverId: string): AlertConfig | undefined` — return config or undefined.
    - `getAllConfigs(): AlertConfig[]` — return `Array.from(this.configs.values())`.
    - `removeConfig(riverId: string): boolean` — delete from both `configs` and `activeAlerts`. Return `true` only if the config existed (use `this.configs.delete()` return value; `activeAlerts.delete()` always runs regardless).

    Import pattern: `import type { AlertConfig, ActiveAlert, RiverData } from './types.js'`

    returning a valid `RiverData` object with all required fields.

    Test groups under `describe('AlertEngine')`:

    `describe('config CRUD')` with tests:

       - create engine, set config (type: 'level', level: 3, enabled: true)

    2. `returns undefined for non-existent config`:

    3. `removes a config and returns true`:

       - verify getConfig returns undefined
    4. `returns false when removing non-existent config`:
       - call removeConfig on unknown riverId
       - expect return value false
    5. `getAllConfigs returns all stored configs`:
       - set 2 configs for different rivers
       - expect length 2

    Actually, the activeAlerts Map is only populated by evaluate() in Plan 02, so for Plan 01, the setConfig cleanup of activeAlerts won't be testable yet. Keep the config CRUD tests focused on what Plan 01 delivers:
    - Basic CRUD operations
    - Config retrieval and listing
