<objective>
Build the engine orchestrator and application entry point with cron-based scheduling.

Purpose: Tie together the DataStore, EventBus, and DatasourceAdapter created in Plan 02 into a cohesive ScraperEngine. The engine handles retry logic (p-retry), stale detection, error state management, and adapter lifecycle. The entry point wires everything together with node-cron scheduling so the system runs autonomously.

Per D-05 — Exponential backoff retry: base 2s, cap 60s, max 3 retries via p-retry.
Per D-06 — Stale-data threshold: 30 minutes since last successful fetch.
Per D-07 — Non-fatal errors retry silently; fatal errors (TypeError from adapter) abort retry via pRetry.AbortError.
Per D-08 — Error state is stored in RiverData.status so Phase 2's UI can surface errors.
Per D-10 — Adapters registered via engine.register(); engine iterates all adapters and consolidates.
Per D-04 — node-cron schedule with env-var-driven cron expression, default every 15 minutes.
</objective>

      - Private properties: `adapters: DatasourceAdapter[]`, `store: DataStore`, `events: ScraperEventBus`, `config: ScraperConfig`
      - Constructor `constructor(config: ScraperConfig)`:
        - Initializes `this.store = new DataStore()`, `this.events = new ScraperEventBus()`, `this.config = config`, `this.adapters = []`
      - Getter `get eventBus(): ScraperEventBus` returning `this.events`
      - Getter `get dataStore(): DataStore` returning `this.store`
      - Method `register(adapter: DatasourceAdapter): void`:
        - Pushes adapter to `this.adapters`
      - Method `scrapeAll(): Promise<void>`:
        - Calls `Promise.allSettled(this.adapters.map(a => this.scrapeOne(a)))`
        - Logs rejected promises to console.error
      - Private method `scrapeOne(adapter: DatasourceAdapter): Promise<void>`:
        - Emits 'status-change' with sourceId and status='fetching'
        - Calls `this.fetchWithRetry(adapter)`:
          - On success: `this.store.update(adapter.sourceId, data)`, emits 'data-update' and 'status-change' (status='ok')
          - On error (caught): checks `this.isStale(adapter.sourceId)` — if stale AND has data, emits 'stale' event with staleSince date. Emits 'status-change' with status='error' and error message.
      - Private method `fetchWithRetry(adapter: DatasourceAdapter): Promise<RiverData[]>` (from Pattern 2):
        - Wraps `adapter.fetch()` in `pRetry()` with config from `this.config.retry`
        - In the retry handler: if error instanceof TypeError (fatal — structural error per D-07), throw `new pRetry.AbortError(error)`
        - Non-fatal errors (Error, etc.) bubble up for retry
        - Uses `onFailedAttempt` callback to console.warn attempt number and retries remaining
      - Private method `isStale(sourceId: string): boolean`:
        - Gets `lastUpdated = this.store.getLastUpdated(sourceId)`
        - If `lastUpdated` is null (no data yet), return false — never stale before first fetch per Pitfall 2 fix
        - Returns `Date.now() - lastUpdated.getTime() > this.config.staleWindowMinutes * 60 * 1000`
      - Private method `staleSince(sourceId: string): Date`:
        - Returns `this.store.getLastUpdated(sourceId)` (already a Date, guaranteed non-null since isStale checks)

    - Export `RiverData` from `./types.js` for convenience if other modules need it.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External API → ScraperEngine (via adapter) | Untrusted external data enters the engine. Validation happens in adapter's parse(). Engine distinguishes fatal (TypeError) vs retryable errors. |
| Engine → DataStore | Internal boundary. Only typed RiverData[] is stored. Engine sets status to 'ok', 'stale', or 'error' per D-08. |
| Engine → EventBus | Internal boundary. Typed events carry structured data. No raw external data crosses this boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Denial of Service | fetchWithRetry() | mitigate | Max 3 retries enforced per D-05. Exponential backoff with 60s cap limits request rate. `Promise.allSettled` prevents one adapter failure from blocking others. |
| T-03-02 | Denial of Service | retry loop on repeated failure | mitigate | Fatal errors (TypeError — structural) trigger `pRetry.AbortError` and skip retry per D-07. Only network/HTTP errors retry. After 3 failed retries, error state is set and engine waits for next cron cycle. |
| T-03-03 | Tampering | Stale data served silently | mitigate | D-06 staleness window (30 min) enforced. Engine emits 'stale' event when last update exceeds window. `status: 'stale'` in RiverData (via D-12) lets Phase 2 UI surface staleness. Pitfall 2 fix: null lastUpdated (no data yet) never triggers stale. |
| T-03-04 | Elevation of Privilege | Event listener injection | accept | EventBus only accepts typed listeners via TypeScript. Same-process only — no external event injection. Phase 2 UI is same Node.js process. |
| T-03-05 | Repudiation | Scrape failures not detectable | mitigate | Every adapter scrape emits 'status-change' with status='ok' or 'error'. Engine logs failures to console.error. Error message is captured in `error?` field on status change. |
| T-03-SC | Tampering | npm dependencies | mitigate | All packages (p-retry, node-cron) verified in RESEARCH.md Package Legitimacy Audit. No postinstall scripts. Approved disposition. |
</threat_model>
